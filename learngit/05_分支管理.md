# 分支管理

## 分支操作

|                   创建切换分支                    |  查看分支  |   合并分支    |     删除分支      |
| :-----------------------------------------------: | :--------: | :-----------: | :---------------: |
| 1. git checkout -b dev<br>2. git switch -c branch | git branch | git merge dev | git branch -d dev |

**场景1**：当在dev分支上做功能，还没有做完就有一个bug需要修复。

> ①利用 git stash 将当前的功能分支的代码储藏起来，先去修复bug，修复完成后合并到master分支。
>
> ②之后再回来dev分支，使用 git stash list 查看储藏的位置，使用 git stash apply stash@{0}（不会删除stash内容，需要再执行 git stash drop） 或者 git stash pop stash@{0}（会直接删除stash内容） 恢复

**场景2**：做完dev的功能后，由于dev是由master拉出来的，所以之前的bug在dev上还有

> 利用git cherry-pick \<commit>，将修复bug的提交剪到dev上即可

**场景3**：一个没有合并过的新功能分支上，已经提交代码了，但是不再需要了

> 强行删除 git branch -D \<branch>

**场景4**：多人协作

1. `git push origin <branch>` 推送自己的修改
2. 如果无法推送， `git pull` 拉取最新的远程的分支，并在本地合并，有冲突则解决完后提交到本地
3. 没有冲突或者解决完冲突后重新push

> 如果git pull提示**no tracking information**，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream-to \<branch-name> origin/\<branch-name>

## git merge 和 git rebase 的区别

核心区别：**是否改变提交历史**

|          维度           |               merge                |  rebase  | 备注                                                         |
| :---------------------: | :--------------------------------: | :------: | :----------------------------------------------------------- |
|      **提交历史**       |               不改变               |   改变   | 无                                                           |
| **是否产生新commit id** |                 会                 |  不一定  | **rebase 会去重写旧的id，但是commit为最新的（刚从其他分支拉出来时）则不会重写** |
|       **可读性**        | 保留分支结构，合并越多，可读性越差 | 结构清晰 | 无                                                           |
